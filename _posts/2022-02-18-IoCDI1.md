## IoC(Inversion of Control)

제어의 역전이라고 하며 개발자가 객체를 관리하지않고 **프레임워크로 객체 관리 제어 권한이 넘어간 것**을 말합니다. 스프링에서는 일반적인 Java 객체를 new로 생성하여 개발자가 관리하지 않고 Spring Container에 모두 맡깁니다.

## DI(Dependency Injection)

`IOC` 가 일어나게된다면 그럼 개발자는 객체를 어떻게 사용하나라는 의문이 들 수 있습니다. 여기서 객체를 사용하기 위해서 외부로부터 객체를 주입받는데 그걸 Dependency Injection, 줄여서 `DI` 라 합니다.

### DI 장점

- 의존성으로 부터 격리시켜 코드 테스트에 용이하다.
- DI를 통하여, 불가능한 상황을 Mock과 같은 기술을 통하여, 안정적으로 테스트 가능하다.
    - Spring Container에 올려야만 (배포해야만) 테스트할 수 있는 부분도 테스팅 가능
- 코드를 확장하거나 변경할 때 영향을 최소화한다. (추상화)
- 순환 참조를 막을 수 있다.
    - 외부에서 들어오기에 자기 자신을 참조하는 일을 예방할 수 있다.

## IoC / DI 예제 코드

base64 또는 url을 encoding 하는 프로그램이 있다고 가정합니다. 테스트를 하려고 하면 코드의 본질을 건드리면서 테스트를 진행해야하는데 이는 매우 비효율적이게 됩니다. (매번 Encoder 생성자의 코드를 바꿔줘야합니다)

```java
public class Encoder {
    private IEncoder iEncoder;

		// base64나 url을 왔다갔다 거리면서 테스트나 실행을 시켜야 원하는 결과가 나온다.
    public Encoder(){
        //this.iEncoder = new Base64Encoder();
        this.iEncoder = new UrlEncoder();
    }

    public String encode(String message){
        return Base64.getEncoder().encodeToString(message.getBytes());
    }
}
```

Encoder라는 클래스를 계속 건들 필요 없이 외부에서 객체를 주입해주는 방법이 DI입니다. Encoder 생성자를 인터페이스 받아오는 형식으로 바꾸고 메인 또한 바꿔줍니다.

```java
package com.company;

import java.util.Base64;

public class Encoder {
    private IEncoder iEncoder;

		// 형식으로 초기화
    public Encoder(IEncoder iEncoder){
        this.iEncoder = iEncoder;
    }

    public String encode(String message){
        return Base64.getEncoder().encodeToString(message.getBytes());
    }
}
```

```java
// main의 출력문
// DI(객체 주입)된 Encoder
// Encoder 객체를 생성할때 base64나 url을 넣어줍니다.
Encoder encoder = new Encoder(new Base64Encoder());
String dResult = encoder.encode(url);
System.out.println(dResult);
```

이렇게 Encoder 입장에서는 외부에서(IEncoder) 생성자를 통해 주입받았습니다. 

또한 개발자가 객체를 주입해줌으로써 관리를 해줄 수 있습니다. (new Base64Encoder)
