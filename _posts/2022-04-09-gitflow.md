## Git Repository 구성 살펴보기

![](https://jihun-24k.github.io/assets/images/git/gitflow1.png){: .align-center}

### 전체적인 3가지 Repository

1. Upstream Repository
    
    개발자들이 협업하는 메인 원격 저장소
    
2. Origin Repository
    
    Forked한 원격 개인 저장소
    
3. Local Repository
    
    내 컴퓨터에 저장되어 있는 개인 저장소
    

### 워크 플로우

Local Repository(개인 저장소)에서 작업을 완료한 한 후 Origin Repository(원격 저장소)로 완료된 작업물을 push합니다.

다음으로 Origin으로 push한 브랜치를 Upstream Repository로 merge하는 Pull Request를 생성합니다. 이 과정에서 반드시 코드리뷰를 거친 후 merge합니다.

마지막으로 다시 새로운 작업을 할 때 Local Repostiory에서 Upstream Repository를 pull하고 작업을 진행합니다.

## Git-flow 전략

### Git-flow의 5가지 브랜치

항상 유지되는 메인브랜치들(**파란색**)과 일정 기간 동안만 유지되는 보조 브랜치들(**빨간색**)이 있습니다.

1. **master**: 제품으로 출시될 수 있는 브랜치
2. **develop**: 다음 출시 버전을 개발하는 브랜치
3. **feature**: 기능을 개발하는 브랜치
4. **release**: 이번 출시 버전을 준비하는 브랜치
5. **hotfix**: 출시버전에서 발생한 버그를 수정하는 브랜치

### 일반적인 Git-flow 개발 흐름

![](https://jihun-24k.github.io/assets/images/git/gitflow2.png){: .align-center}

- develop 브랜치에서는 상시로 버그를 수정한 커밋들이 추가됩니다.
- 새로운 기능 추가 작업이 있는 경우 develop 브랜치에서 feature 브랜치를 생성합니다.
- feature 브랜치는 언제나 develop 브랜치에서 시작됩니다.
- 기능 추가 작업이 완료됐다면 feature 브랜치는 develop 브랜치로 merge 됩니다.
- develop에 이번 버전에 포함되는 모든 기능이 merge 됐다면 QA를 하기 위해 develop 브랜치에서부터 release 브랜치를 생성합니다.
- QA를 진행하면서 발생한 버그들은 release 브랜치에 수정됩니다.
- QA를 무사히 통과했다면 release 브랜츠를 master와 develop 브랜치로 merge합니다.
- 마지막으로 출시된 master 브랜치에서 버전 태그를 추가합니다.
